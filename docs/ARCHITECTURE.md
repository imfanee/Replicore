# Replicore — Architecture

This document explains how Replicore works so operators and engineers can reason
about its behavior. It is product-facing; it describes the shipped engine, not a
plan.

## 1. Model

Replicore is **multi-master**: every node owns a local copy of a shared file tree
(`share_dir`) and accepts local writes at full speed. The engine observes those
writes and propagates them to every peer. There is no primary, no quorum on the
write path, and local I/O is never blocked on the network.

Consistency is **eventual**: after writes quiesce and peers are reachable, every
node converges to a byte-identical tree (including conflict copies). Convergence
is a property of the data structures, not of timing.

```
   node A                        node B                        node C
 ┌────────┐  op-log + chunks   ┌────────┐  op-log + chunks   ┌────────┐
 │ share  │ <================> │ share  │ <================> │ share  │
 │  dir   │   QUIC / mTLS      │  dir   │   QUIC / mTLS      │  dir   │
 └────────┘                    └────────┘                    └────────┘
   watcher                       watcher                       watcher
   + rescan                      + rescan                      + rescan
```

## 2. Two replication channels

**Metadata travels as an operation log.** Every observed change (create, modify,
rename, delete, metadata change) becomes an **op** appended to a local,
WAL-backed op-log (`rusqlite`). An op records the path, the change kind, the
file's content hash, a canonical metadata snapshot, the file UUID (identity), and
a **version vector** stamp. Peers subscribe to each other's op streams.

**File data travels as content-addressed chunks.** A file's content is split with
**FastCDC** into variable-size chunks, each named by its **BLAKE3** hash. A file
is described by a *manifest* (an ordered list of chunk hashes). To materialize a
file a node fetches only the chunks it is missing, from any peer that has them
(multi-source). Identical content anywhere in the mesh is stored and transferred
once.

## 3. Causality: version vectors, never the clock

Each node keeps a per-origin counter. A version vector (VV) is the set of
counters a version has observed. Comparing two VVs yields exactly one of:
**Dominates**, **Dominated**, **Equal**, or **Concurrent**.

- If version X dominates Y, X is strictly newer — apply X, no conflict.
- If X and Y are concurrent, that is a genuine conflict.

Wall-clock time is **only** used to break ties between already-concurrent
versions; it never establishes causality. This is the single most important
invariant in the system: a clock skew can never cause a newer write to lose to an
older one.

## 4. Conflict resolution (deterministic, no loss)

When concurrent versions exist for a path, Replicore computes the **maximal
antichain** of candidates (the set of mutually-concurrent versions, with any
dominated version discarded). Pairwise resolution is *not* used — it is
non-confluent and can pick different winners on different nodes. The antichain is
ordered by a deterministic key `side_key = (kind_rank, content_hash, meta_hash)`;
the top element is the **winner** and is applied in place. Every loser is written
beside it as a **conflict copy** with a deterministic, node-agnostic name derived
from the loser's content hash and a stable metadata subset.

Properties (proven and property-tested):

- **Deterministic / confluent:** every node, regardless of the order in which it
  saw the versions, computes the same winner and the same set of copy files.
- **No silent loss:** the only versions that do *not* get a distinct copy are
  those identical in both content and durable metadata to one already kept.

See `docs/review-3a-conflict.md` and `docs/review-copy-naming.md` for the proofs.

## 5. Atomic apply

A replicated file is never exposed half-written. Apply is:

1. stage the content into a temp file **in the destination directory**,
2. `fsync` the staged file,
3. verify its BLAKE3 hash against the manifest,
4. `rename(2)` it over the target (atomic on POSIX),
5. `fsync` the parent directory.

Metadata is applied in a fixed order (xattrs → ownership → mode → mtime) so that,
e.g., a `chown` that clears setuid is followed by the `chmod` that restores it.

## 6. Loop prevention

A naive mesh storms: a node applies a peer's write, its own watcher sees the
apply, and it re-broadcasts. Replicore prevents loops with **two** independent
mechanisms, both required:

1. **Version-vector dedup** — an op already covered by the local VV is ignored.
2. **Apply-suppression** — the filesystem event generated by the engine's own
   apply is recognized and not turned into a new op.

Either alone is insufficient.

## 7. Detecting local change

Two sources feed the op-log:

- **fanotify (FID reporting)** — low-latency notification of local writes. It is
  *best-effort*.
- **Periodic Merkle rescan** — walks the tree, recomputes subtree hashes, and is
  the **authoritative** backstop. It catches anything the watcher missed
  (notably writes arriving via `nfsd`, which fanotify does not reliably deliver).

The rescan is never weakened on the assumption the watcher is complete.

## 8. Anti-entropy (self-healing)

Periodically, and on connect, two nodes run an anti-entropy session. They
exchange a Merkle summary of their trees, descend only into differing subtrees,
and reconcile via the op-log. A node never applies a peer's *live* ops before it
has completed an anti-entropy session with that peer (the reconcile gate) — this
guarantees a consistent starting point before streaming.

## 9. Deletes are tombstones

A delete becomes a tombstone op, not a hard unlink of replicated state.
Tombstones propagate like any other op and are garbage-collected only after all
peers have acknowledged them and a safety window has elapsed. A late write can
never resurrect a deleted file.

## 10. Cluster membership and the control plane

Membership is dynamic. Two configurations exist, each with exactly one owner:

- **Intent** (`replicore.toml`) is **human-owned**. The daemon never writes it.
  Its `[[peers]]` list is the seed/bootstrap set.
- **Roster** (`roster_path`) is **daemon-owned**. Dynamically-learned members
  live here.

Membership is an **epoch-versioned register** that converges deterministically
(merge winner = `max(epoch, rank(kind), blake3(canonical bytes))`; the tie-break
hashes canonical bytes, never a signature). Concurrent add/remove from different
nodes converge to the same result on every node.

**Announcement is not authorization.** A peer enters the data path only after its
certificate validates against the pinned trust anchor. Membership changes are
**admin-signed**: `replicorectl member add/remove` signs the entry client-side
with the cluster admin key; the daemon never holds that secret. Being
vouched-for by a peer is not trust.

A node joining a live cluster bootstraps from a snapshot frontier and resumes the
live op stream from a **version-vector frontier**, so bulk bootstrap and parallel
live operations compose with no lost or double-applied ops.

## 11. Transport and flow control

All peer traffic is **QUIC** (`quinn`) over **mutual TLS** with a pinned
peer-certificate allowlist. Congestion control is **BBR** (chosen over Cubic for
WAN throughput under loss). Bandwidth is shaped by a debt-model token-bucket
system: a global budget intersected with a per-peer budget, with priority lanes
for small/interactive assets and an optional time-of-day schedule.

## 12. Production safeguards

- **Free-space guard** — the engine reserves `max(reserve_bytes, reserve_percent
  of capacity)` and stops accepting inbound data rather than filling the disk.
- **Metrics** — Prometheus `/metrics`; liveness/readiness via `/healthz`.
- **Crash recovery** — manifests persist atomically; on restart the scanner
  re-attributes orphaned-but-correct staged files instead of re-fetching.

## 13. Wire/state versioning

The protocol is a flag-day version: the current mesh runs ALPN `replicore/4`
(PROTO v4). The whole mesh upgrades as a unit; mixed-version meshes are refused at
the handshake. The Merkle leaf and op record formats are versioned alongside it.
